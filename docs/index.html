Cruller: Bun's Zig Runtime, Continued on Zig 0.16

Cruller 是一個基於 Bun 舊版 Zig 發行的分支，專注於執行現有的生產型 JavaScript 伺服器，並已移植到原生 Zig 0.16。它保留了運行時所需的關鍵組件，例如 JavaScriptCore、Bun.serve 和網路功能，但移除了開發工具如套件管理器和測試執行器，使其成為一個輕量級的運行時。此項目旨在作為運行時而非完整的 Bun 替代品，與官方 Bun 相比，其大小減少了約 18%，且性能表現相當。 - 專為預先建置的 JavaScript 服務器優化，大幅削減開發相關功能。 - 支援原生 Zig 0.16 和 Linux x64，具備尺寸和部署優勢。

「專注於生產的運行時」對於 Cruller 來說具體意味著什麼？

這表示 Cruller 旨在提供一個精簡、高效的環境，專門用於執行已經建置好的 JavaScript 服務器。它移除了如套件管理器、打包器和測試執行器等開發工具，只保留了運行生產服務所需的最小核心功能，從而降低了資源消耗並簡化了部署。

對於 Cruller 的目標使用場景，它與完整的 Bun 運行時相比主要優勢是什麼？

Cruller 的主要優勢在於其更小的體積和更低的資源消耗，實現了約 18% 的尺寸縮減。這使得它更適合部署在資源受限的環境或需要快速啟動的場景中。此外，由於其專注於核心運行時功能，也可能帶來更簡潔的運行時行為和潛在的穩定性優勢。

支援「原生 Zig 0.16」對於 Cruller 的開發和未來發展有何重要意義？

支援原生 Zig 0.16 意味著 Cruller 不再依賴 Bun 舊版中帶有補丁的 Zig 編譯器集成，而是採用了標準的 Zig 工具鏈。這簡化了構建圖並提高了項目的可維護性、兼容性和可移植性，使其更容易跟隨 Zig 語言本身的發展，也為潛在的社區貢獻和跨平台支援奠定了基礎。

Amiga 1000: Ten years ahead of its time

Amiga 1000 電腦於 1985 年推出，被譽為領先時代十年。它擁有先進的自訂晶片組，提供彩色圖形、立體聲、圖形使用者介面，尤其其全搶佔式多工處理能力，在當時遠超同期的 Apple Macintosh 和 Atari ST，甚至早於 Windows 95。儘管技術卓越且性價比相對較高，但由於 Commodore 糟糕的行銷和管理，未能充分發揮其潛力。 * 主要優勢：先進的全搶佔式多工處理。 * 主要瓶頸：Commodore 缺乏有效的產品行銷和領導。

Amiga 1000 的「全搶佔式多工處理」與其他系統相比有何特別之處？

「全搶佔式多工處理」允許作業系統完全控制 CPU 的時間分配，確保每個程式都能公平運行，即使一個程式卡住也不會影響其他程式。這與當時許多作業系統（如 Windows 3.0/3.1 和 Apple System 7）採用的協同式多工處理不同，後者需要程式主動放棄 CPU 控制權，更容易導致系統不穩定。

既然 Amiga 1000 技術如此先進且價格合理，為何它未能主導市場？

Amiga 1000 未能主導市場主要歸咎於 Commodore 公司糟糕的行銷策略和領導層。他們未能有效地向大眾傳達 Amiga 的獨特優勢和未來願景，也缺乏像 Apple 那樣吸引人的產品敘事。此外，公司內部管理不善，領導人更專注於從公司中謀取利益而非長期發展。

後來的個人電腦或作業系統吸收了 Amiga 的哪些創新或改進？

Amiga 1000 的全搶佔式多工處理能力直到 1995 年 Windows 95 推出後才被主流 PC 作業系統普遍採納。文章指出，作者在使用 Windows PC 時，甚至到 1994 年仍感覺其穩定性不如 Amiga，特別是在同時進行多個任務時。Amiga 在圖形和聲音方面的早期領先也為未來多媒體PC的發展奠定了基礎。

這些變革會不會讓新的或經驗較少的安全研究人員更難參與或獲得獎勵？

雖然公開計畫的獎金有所降低且有訊號要求，但 GitHub 表示這「不是新手研究人員的障礙」。HackerOne 平台為未達標準的研究人員提供最多四次初始提交機會，足以讓真正有發現的新手展示其技能，並有機會進入 VIP 計畫。

除了獎勵金重組和 VIP 計畫外，GitHub 還計劃採取哪些措施來改善與安全研究社群的互動？

GitHub 還將投入更多資源以縮短響應時間、提供更清晰的嚴重性判斷理由，並加強社群參與。他們計劃透過參與 DEFCON 等會議和持續的外展活動，直接與研究人員互動，建立更緊密的合作關係。
